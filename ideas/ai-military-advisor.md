# 军议幕僚 + 将领传记

> 一个把 "AI Native" 从开发模式升级为玩法卖点的方向

**创建**: 2026-04-08
**状态**: 创意草案（未立项）
**作者**: plancklin + Claude 策划会
**定位**: Steam 商业化方向的候选玩法

---

## 一句话

> 你是战场上的将军，你有一位会吐槽的 AI 军师。你们一起打仗、一起争论、一起被载入史书。将军战死即永别，但他的传记会留下。

---

## 背景与动机

项目现有的玩法愿景（`docs/design/game/gameplay-vision.md`）已经有很好的差异化骨架：

- **将领视角**：玩家不是上帝，而是战场上的一位将领
- **人海阵型 + 溃败传染**：规模感与士气崩溃的视觉冲击
- **战策时刻**：暂停 = 指挥权，而非减速

这些已经让游戏区别于市面上大部分 RTS。但如果要追求 Steam 商业化上的"爆款相"，还需要一个**只有这个项目能做**的独特卖点——利用已经投入巨大成本的 AI Renderer / Sensor Registry 基础设施，把 LLM 做成游戏内 NPC，而不只是开发助手。

这就是**军议幕僚系统**的起点。

## 五个脑洞的商业评估

五个脑洞原始讨论详见讨论记录。按 Steam 爆款维度打分：

| 维度 | 含义 |
|---|---|

| **生态位空白度** | Steam 上有没有对手？ |
| **重玩性引擎** | 玩家会不会玩 100 小时？ |

|---|:-:|:-:|:-:|:-:|:-:|:-:|
| #1 军议幕僚 | ★★★★★ | ★★★★ | ★★★★★ | ★★★ | ★★★★ | **21** |
| #2 信息战 | ★★★ | ★★ | ★★★ | ★★★★ | ★★★★ | 16 |

| #4 情绪谱系 | ★★ | ★★★ | ★★ | ★★★ | ★★★★★ | 15 |
| #5 将领传记 | ★★★★ | ★★★★ | ★★★★ | ★★★★★ | ★★★★ | **20** |

> **五个脑洞全文**：
> 1. **军议幕僚**：LLM 作为游戏内军师，战前献策、战中吐槽、战后评语
> 2. **信息战**：你看到的敌方兵力是斥候估算，不是真实值；可下达疑兵之计
> 3. **沙盘战策**：暂停时画面变成古代沙盘，像下棋一样画行军箭头
> 4. **情绪谱系**：士气不是条，是恐惧/狂怒/疲惫/信念的情绪谱系
> 5. **将领传记**：永久死亡 + LLM 自动生成战役传记，积累成一部战争史

## 推荐方案：#1 + #5 组合

**最有爆款相的不是任何单独的脑洞，而是军议幕僚 + 将领传记的组合。**

### 为什么是这两个

1. **共用同一条技术管道**：都是 "LLM 读 AI Renderer 导出的结构化战场状态 → 产出文本"。做 #1 等于做了 #5 的 80%
2. **情感上互补**：#1 是战中陪伴，#5 是战后沉淀。一个是刺激，一个是回味
3. **只有你们能做**：对手即使看到也抄不走——抄 LLM 对话容易，抄一套从第一行代码就为 AI 可读而设计的架构不容易。这是"AI Native"的真正护城河
4. **低美术成本的差异化**：两人小团队最怕走美术竞赛，这两个方案的差异化在交互和叙事上
5. **天然 Roguelike 循环**：将领出战 → 幕僚伴你征战 → 战死 → 传记载入史册 → 换新将领开下一局

### 玩家体验骨架

```
┌──────────────────────────────────────────────┐
│  战役开始                                     │
│  ├─ 选一位幕僚（不同性格）                     │
│  └─ 战前对话：幕僚分析局势 + 献策                │
│           ↓                                   │
│  战役进行中                                    │
│  ├─ 人海阵型 + 溃败传染（Phase 15-17 成果）    │
│  ├─ 战策时刻可召唤幕僚（读当前 snapshot 建议） │
│  └─ 将领可死亡（永久）                         │
│           ↓                                   │
│  战役结束（胜/败/将领阵亡）                    │
│  ├─ LLM 读战后 snapshot → 生成一页纸传记       │
│  ├─ 存入"史册"                                │
│  └─ 开新将领 → 下一局                          │
└──────────────────────────────────────────────┘
```

---

## 为什么能对上 2026 年 Steam 的口味

- **"AI 在游戏里"题材有时代红利但还没有代表作**。玩家好奇但没见过好东西，谁先做出"AI 让游戏真的变得更好玩"的代表作，谁就拿到这个标签
- **Roguelike + 叙事**在 Steam 是稳定长青品类（Hades、Slay the Spire、Darkest Dungeon）
- **"让 AI 讲故事"**一直有人想做但没做对——别人是"事后硬套 LLM"，你们是"从一开始就把游戏状态结构化"。这个差距是三年的工程债
- **古代战争**赛道最近只有 Manor Lords 火了，RTS 向的古代战争有空档
- **两人小团队 + 差异化叙事**是 Steam 最容易出黑马的组合（ConcernedApe / Balatro / Lethal Company 都是这套）

## 前三个里程碑（建议）

### M1 — 垂直切片：最小闭环
**目标**：验证"玩家会因为 AI 军师而想再打一场"这件事成不成立。

- 只做一场预设战役（固定地图、固定敌军、固定起手兵力）
- 有将领、有人海阵型、有溃败传染（复用 Phase 15-17 成果）
- 植入一个最小幕僚：
  - 战前一段对话（"敌在东岭集结，主公当如何？"）
  - 战中按键召唤一句建议
- LLM 走本地小模型或 API，先不纠结成本

**验证问题**：玩家会不会在对话结束后主动说"我想再来一次，听听它这次说什么"？

### M2 — Roguelike 循环闭合
**目标**：让"将军死亡 → 传记 → 新将军"这个循环跑起来。

- 加入将领永久死亡
- 加入战后自动生成的一页纸传记（LLM 读 snapshot 生成 200 字左右）
- 加入"史册"画面：历代死去将领名录，可点开看各自传记

**验证问题**：玩家会不会因为"不想让这个将军死"而更谨慎地打仗？会不会主动翻看上一场的传记？

### M3 — 幕僚人格化
**目标**：让幕僚从"一个 LLM"变成"有性格的角色"。

- 做三位不同性格的幕僚（勇将型 / 谨慎型 / 阴险型），战前可选
- 同一局面，三位幕僚给出不同建议
- 战后传记中会提到幕僚的功过

**验证问题**：玩家会不会说"这局我选了那个阴险的，结果他建议我疑兵之计，真有意思"？

> 每个 M 结束后做一次 20 人小规模闭门测试（朋友圈 / Reddit 小圈子），看反应决定要不要继续。

## 风险与未解问题

| 风险 | 说明 | 可能应对 |
|---|---|---|
| **LLM 延迟** | 战策时刻召唤幕僚，如果等 3 秒回复，沉浸感就断了 | 预生成 / 本地小模型 / 异步后台生成多个候选 |
| **LLM 成本** | 如果走 API，长期运营成本高 | 优先支持本地模型；云 API 作为高质量档位 |
| **联网劝退** | 部分 Steam 玩家抗拒"必须联网"的游戏 | 本地模型兜底，联网是可选增强 |
| **LLM 建议不够聪明** | 玩家会很快发现幕僚在"说废话" | 幕僚的建议要基于结构化 sensor 数据做 rule-based 过滤，LLM 只负责包装成人话 |
| **战后传记枯燥** | 千篇一律的"某年某月，公率兵……"会让人失去兴趣 | 多样化模板 + 关键事件抽取 + 文风随幕僚个性变化 |
| **核心玩法未稳定就铺 AI** | 如果战场本身不好玩，再好的 AI 军师也救不了 | 必须先让 Phase 15-17 的人海战场真的好玩，再做 M1 |

---

## 不做什么

为了不被诱惑跑偏，明确列出**这个方向不要碰**的东西：

- ❌ **不做 AI 生成美术**：美术走朴素低面数风格或水墨风，把 AI 预算全部投到玩法交互上
- ❌ **不做联机对战**：项目宪章已经决定绕开网络同步
- ❌ **不做开放大地图战略层**：维持"一场战役 = 一个封闭战场"，避免战略层复杂度爆炸
- ❌ **不做 LLM 自由对话**：幕僚的交互是**选项式**的（玩家选问题/指令，LLM 在约束下生成回答），不给玩家无限对话框
- ❌ **不做"AI 生成关卡"**：地图在 M1/M2 阶段是固定的，不追 PCG

## 最后的策划叮嘱

**爆款游戏几乎都不是追着爆款目标做出来的**。Balatro 的作者只是想做一个自己喜欢的 Roguelike；Stardew Valley 的作者只是想做一个他童年喜欢的农场游戏。爆款是因为他们把自己真心喜欢的那个东西做到了极致，然后运气站在了他们这边。

所以真正的建议是：**用"军议幕僚 + 传记"作为设计方向，但把目标改成"做一个我和竹晓自己会爱玩的小游戏"**。爆款作为副作用发生的概率，反而比硬追它更高。

项目宪章的成功标准写着"过程有趣、学到东西、产出一个能玩的东西"——这个方向恰好同时满足这三个，而且顺带撞上了一个可能爆款的生态位。这已经是最好的组合。
