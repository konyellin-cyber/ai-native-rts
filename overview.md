## 项目定位

| 维度 | 定义 |
|------|------|
| **定位** | 娱乐/生活项目，和竹晓一起探索游戏开发 + AI 辅助 |
| **成功标准** | 过程有趣、学到东西、产出一个"能玩"的东西 |
| **失败条件** | 过程不快乐 → 直接换方向或放弃，不硬撑 |
| **时间投入** | 每周 5-10 小时（下班后/周末），不追求速度 |
| **AI Native 定义** | AI 是编码助手（写代码、找 bug、跑测试），人是玩法设计师 |

**与其他项目的关系**：ai-sidetouch / wedata / 记忆系统 → 工作技能，优先级更高；ai-native-rts → 生活乐趣，可以慢，可以停。

## 游戏设计

```
资源:  1 种（水晶）
建筑:  2 种（基地=生产单位，矿点=采集目标）
单位:  2 种（工人=采集，战士=战斗）
胜利:  摧毁对方基地
```

**扩展路径**：MVP → 科技树 → 第 2 种族 → 地形 → 战役

## AI 协作模式

```
两人讨论设计/决策
       ↓
  一人操作 AI 实现
       ↓
    验证通过/问题
       ↓
    （循环）
```

**AI 渲染器 = AI 自主 debug 的眼睛**：没有结构化数据，AI 只能猜；有 AI 渲染器，"第 47 帧 pos(23,301) vel 反转" → 精准定位，一次修复。详见 [`docs/design/tech/ai-renderer.md`](./docs/design/tech/ai-renderer.md)。

## 关键决策记录

| 决策 | 结论 | 理由 |
|------|------|------|
| 引擎选型 | Godot 4.6 | 全 CLI 可跑，headless 支持好，AI 友好 |
| 网络模式 | 人 vs 简单 AI，不做联机 | 网络同步是巨坑，MVP 阶段绕开 |
| 验证方式 | headless 优先，窗口仅看视觉 | 程序化验证可重复、可自动化 |
| AI Renderer 共享化 | symlink 到 `shared/` | 避免多 Phase 之间代码复制漂移 |
